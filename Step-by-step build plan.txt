what will not be included yet

Docker

Not yet

Deliverable

A clear project definition so the build stays focused.

Step 2: Design the API contract

Before coding logic, we define the backend contract.

We will design:

GET /health

POST /chat

POST /ingest

GET /sessions/{session_id}

DELETE /sessions/{session_id}

Most important

We finalize the response shape for /chat.

Example:

{
  "answer": "Order 1024 is currently in transit.",
  "action": "tool",
  "used_tools": ["order_lookup"],
  "sources": [],
  "memory_summary": "User is asking about delayed order 1024.",
  "escalation": {
    "needed": false,
    "reason": null
  }
}
Deliverable

Stable request/response schemas.

Step 10: Build session memory

Now we add remembering.

We will implement:

session-based chat history

store messages by session_id

optional memory summary

session reset route

Goal

Follow-up questions should work.

Example:

“My order is 1024.”

“Can you check its status?”

The second one should understand what “its” means.

Deliverable

Working memory layer.

Step 11: Build the routing logic

Now we teach the system how to choose the correct path.

It should decide:

use RAG

use tool

escalate

maybe direct answer for simple greetings

First version

Start simple:

keyword/deterministic rules

optional LLM-assisted classifier later

Example:

“refund policy” → RAG

“where is order 1024” → tool

“I want to file a fraud complaint” → escalate

Deliverable

A routing service that selects action type.

Step 12: Build the main chat orchestration service

This is the brain of the backend.

Flow:

receive message

check escalation

build final structured response

save memory

Deliverable

One clean chat service for /chat.
